## April 24 — Prompt 9: openFDA dynamic info (F7)

Three-tier hybrid lookup so any medication grandma adds gets info without needing a JSON edit: Tier 1 = curated drug_info.json (fast, offline, grade-6 prose). Tier 2 = on-disk LRU cache of prior openFDA fetches (50-entry cap, fast, offline, survives relaunches). Tier 3 = live `https://api.fda.gov/drug/label.json` query on full miss. If everything misses, the friendly "ask your pharmacist" copy stays.

`lookupAny(for:) async throws -> DrugSource` is the orchestrator. Throws only when Tier 3 fails *and* there's no T1/T2 fallback, so offline-first is preserved for known meds. Cached entries get a "(cached)" suffix on the source label. Honest "From openFDA (clinical label)" badge for dynamic content + italic per-field caption "This is the official FDA label. Ask your pharmacist if any of this is unclear" — no pretending FDA prose is grade-6.

Commit `8bc2ef7`. 33/33 tests (11 repo + 11 curated tier-1 + 5 three-tier flow + 6 cache LRU).

Then a UX fix: dynamic content was rendering as walls of clinical prose instead of the bullet layout the curated meds get. Cause: drug_info.json provides arrays; openFDA returns flat strings. Built `OpenFDAContentParser` with a four-strategy waterfall — line bullets first (with the smart `*` / `-` requires-whitespace rule so "well-being" doesn't get split), then numbered lists via NSRegularExpression, then semicolons, then introducer-plus-comma (`include`, `such as`, `may include`, `are`, `consist of`) with a `nonItemPrefixes` filter that drops trailing clauses like "were reported." Both branches now route through the same `BulletRow` view — visual parity guaranteed by construction. 18 new parser tests on top of the 33.

Lesson: when one source returns prose and another returns arrays, parse to a common shape before rendering — don't fork the view.

## April 25 — Camera scan + Vision OCR

VNRecognizeTextRequest with the `.accurate` revision. AVFoundation camera with the proper permission gates (`.unconfigured`, `.denied`, `.failed`, `.authorized`). PrescriptionParser as a regex chain — name from the top-most alpha line (after filtering pharmacy banners like "pharmacy", "drugstore", "rx number"), dose with `\b(\d+(?:\.\d+)?)\s*(mg|mcg|g|ml|iu|units?)\b`, frequency mapping, food rule, quantity (explicit `Quantity:` label first, falls back to bare `<N> tablets` only when N≥2 digits and isn't preceded by `Take` so "Take 1 tablet" doesn't hijack the quantity).

The 17 parser tests caught three real bugs in one run: "twice daily" was matching as "Once daily" because the higher-arity patterns required the literal word "day" but `\bdaily\b` was on the once mapping (fixed: `(?:day|daily)` on twice/three/four), the name parser was picking pharmacy banners over the actual drug name (fixed: filter list), and "Take 1 tablet" was being read as quantity 1 (fixed: lookbehind for `Take`).

Surface: a "Scan a prescription label" button on AddMedicationFlow Step 1 above the manual TextField, dot-and-italic confidence indicators on the review screen — green check for ≥0.8, amber triangle for 0.5–0.8 ("Please double-check"), red octagon for <0.5 ("We couldn't read this — please type it in"). Low-confidence fields stay blank in `applyScanned` so the user types them — never put words in the doctor's mouth (B.1 S2). `INFOPLIST_KEY_NSCameraUsageDescription` baked into Debug + Release configs (Info.plist is autogenerated; without the key the app crashes on first camera access).

Simulator can't capture frames so the camera-permission gate path was the only one I could exercise from the shell. Real-iPhone scanning of an actual prescription bottle is the canonical test — pending.

## April 25 — Prompt 12 attempted: Punjabi localization (interrupted)

Started full Gurmukhi localization — Localizable.strings en + pa, drug_info_pa.json mirror, language picker on first launch, AVSpeech `pa-IN` voice helper. Caught a critical mismatch mid-prompt before any commit: my grandparents speak Punjabi but **don't read Gurmukhi**. The script most Indian-Punjabi Sikhs use is Gurmukhi; many Punjabi-speaking elders are literate only in English (or in Shahmukhi if they're from Pakistan), having migrated as adults.

Pressed Esc on Claude Code before the big refactor committed. The right path is probably audio-first — translate spoken output (notification voice readouts) to Punjabi while keeping the UI in English (or simplified English). Or possibly Hindi/Devanagari if that's what they actually read. Need to ask the family before re-running this prompt.

Critical research finding from the camera-scan run that's relevant whenever Punjabi gets re-attempted: Apple's Vision framework does NOT support Gurmukhi for OCR in any revision through iOS 18. Supported scripts are English, French, Italian, German, Spanish, Portuguese (PT-BR), Chinese, Cantonese, Japanese, Korean, Russian, Ukrainian, Arabic, Thai, Vietnamese, Swedish. Two fallbacks exist (Tesseract on-device, cloud OCR) but both break the offline-first rule. Recommendation logged: leave OCR English-only; pharmacy labels in BC are in English anyway.

Lesson, hard one: ask which **script** the user reads, not just which language they speak. Localization assumptions can waste a whole prompt's worth of work.

## April 25 — Face ID crash on real iPhone (commit `43fdbbb`)

App hard-crashed on the first tap of "Use Face ID" on the login screen. Smoking gun was `INFOPLIST_KEY_NSFaceIDUsageDescription` missing from both Debug and Release target configs — only the camera key had been added when the Auth prompt landed. iOS hard-kills the process on the first `LAContext.evaluatePolicy(.deviceOwnerAuthenticationWithBiometrics, …)` call when this key is absent — the kernel doesn't even let the completion handler fire. That's the real-iPhone crash exactly.

Two secondary tightenings while I was in there: `AuthService.biometricLogin()` was throwing silently (the LoginView catch swallowed errors without setting `errorMessage`), so every throw path now sets a localized AuthError description. Re-tagging do/catch ensures raw `LAError` enums never leak to the UI. Added `[AUTH-DEBUG]` logging at the start of `biometricLogin()` printing biometryType + canEvaluate + LAError code, with a static `describe(_:)` helper for human-readable type names.

Side-finding worth flagging separately: I'd renamed the bundle ID in Xcode from `com.placeholder.dosely` → `com.medication.dosely`. That means the Firebase iOS app needs its bundle ID updated to match (or `FirebaseApp.configure()` will silently fail to bind). Also `Keychain.swift`'s `kSecAttrService` is still hardcoded to the old name — opaque partition string, works as-is, but stale.

Lesson: every Info.plist usage description is a separate kernel-level gate. Adding NSCameraUsageDescription doesn't grant Face ID; each capability needs its own key.

## April 25 — Bug list from real-iPhone testing

Three more issues surfaced while testing on the actual phone, all queued for fixes:

Dark/light mode invisible text — DSColors uses single hex literals so tokens don't adapt. In Dark Mode the dark `dsTextPrimary` (#1A202C) shows on top of the system's near-black background and becomes unreadable. Fix queued: rewrite each token with `UIColor(dynamicProvider:)`, with dark variants chosen for ≥4.5:1 contrast (e.g., `dsPrimary` lifts #2B6CB0 → #4A90E2 because pure #2B6CB0 fails contrast on near-black). Plus a contrast-test suite using WCAG luminance.

Face ID setup prompt missing after sign-up — the alert from Prompt 7's spec ("Enable Face ID for quick access next time?") either was never wired or is silently gated. Queued.

Face ID always errors with "Your session has expired. Please sign in with your password" — this is the architectural one. Original Prompt 7 design assumed we could mint a new Firebase ID token from a stored refresh token after biometric success, but `Auth.auth().signOut()` wipes Firebase's own keychain entries, so post-logout there's nothing to unlock. The right pattern is 1Password/banking-style: split "Firebase signed-in" from "Dosely locally locked." Default Sign Out becomes "lock the app, keep Firebase session"; new "Sign out completely" wipes everything. Face ID success just flips the local lock — it doesn't touch Firebase. Queued.

Camera could use multi-photo or video capture for wrapped labels — pill bottles wrap text around the cylinder, single capture misses the back. Queued.

Lesson, third time in this project: error collapsing hides bugs. "Your session has expired" was hiding a fundamental design flaw. Distinct error codes everywhere from now on.

## May 1 — Pull-to-refresh on Today, History, People (commit `880313e`)

The app's been listener-driven for fresh data since Prompt 16. When a listener drops — network blip, backgrounding, anything — the user has no recovery short of force-quitting. Native pull-to-refresh fills that gap: an obvious gesture that runs the same per-collection reads as the listener pipeline, but as a one-shot round trip.

`SyncCoordinator.refresh()` is the new entry point. Concurrent `async let` fetches over careCircle, people, medications, schedules, and dose logs, then the existing private mirror helpers upsert into Core Data and prune locally-orphaned rows. Aborts on the first failure rather than mirror a partial snapshot — mistaking "couldn't ask" for "the server has empty data" would let the orphan-pruning helpers wipe the local cache. Errors land as `SyncRefreshError` (offline / permissionDenied / unknown) so the UI maps each to distinct copy.

That cache-wipe-on-empty trap also hid in `FirestoreService.fetchPeople`, which used to return `[]` when the SDK wasn't configured. Hardened to throw `.offline` instead, alongside three new fetchers — `fetchMedications`, `fetchDoseSchedules`, `fetchDoseLogs` — that follow the same contract.

UI glue lives in `Dosely/Features/Common/PullToRefresh.swift`: a `PullToRefresh.perform(messageBinding:)` helper runs the refresh and writes any error's localized copy to a binding, plus a `pullToRefreshBanner` view modifier that renders an inline banner and auto-dismisses it after ~3s. `TodayView`, `HistoryView`, `PeopleManagementView`, and `SupervisorDashboardView`'s todayTab all wrap their content in a ScrollView with `.refreshable` and the banner modifier. Pull-to-refresh works for primary and secondary supervisors equally — the gating already lives in `SyncCoordinator`, which respects circle membership regardless of role.

Two `TodayView` shape changes worth noting. The outer container is now a single ScrollView; the date header used to live in a VStack outside the content's conditional inner ScrollView, which meant pulling from the header didn't trigger refresh. And the populated state's inner ScrollView became a plain `LazyVStack` so the gesture isn't fighting a nested scroll view.

Tests in `DoselyTests/PullToRefreshTests.swift`: silent no-op when there's no active circle, `.offline` against an unconfigured `FirestoreService`, mirrors fresh data into Core Data against the local emulator. Three smoke tests host each tab in a `UIHostingController` and assert a `UIScrollView` is present in the rendered hierarchy. SwiftUI's `.refreshable` attaches its `UIRefreshControl` to the underlying scroll view; without it the gesture has nowhere to live, and a refactor that wraps content in a VStack instead would silently break it.

The trap I keep hitting on this codebase: a function that "gracefully" returns an empty result when something's wrong reads identically to a function that says "yes, the answer is empty." Combine that with downstream code that prunes anything not in the result and you have a cache-wipe machine. Better to throw and let the caller decide.

## May 1 — Add-person chooser was rendering raw localization keys (commit `7541904`)

People → "+" → "Add a family member" was showing `supervisor.add.managed.title` etc as the option titles. The user's first guess was missing keys; the keys were actually present in both `en.lproj` and `pa.lproj`. Bug was at the call site in `Dosely/Features/Supervisor/People/AddPersonFlow.swift`:

    Text(type.titleKey)   // type.titleKey is a runtime String

`Text(_:)` on a runtime `String` binds to the verbatim initializer and skips Localizable.strings. Only `Text(LocalizedStringKey)` — which Swift picks for string literals — routes through the bundle. Same trap was sitting on the `.fillForm` nav-title branch, latent until anyone made it to step 2 of the flow. Fix was three `L(...)` wrappers.

Two tests in `DoselyTests/AddPersonFlowLocalizationTests.swift`. The first probes the lookup table — every chooser key must resolve to something other than itself in `en.lproj` — which catches the user's original hypothesis if a future chooser type ships without strings. The second renders the chooser in a `UIHostingController` and walks the UIKit hierarchy for any `UILabel.text` that equals one of the six raw keys. That one catches the actual bug class (key resolves but the call site never asked for localization), which the lookup test alone would miss.

The diagnosis matters more than the fix. The user's hypothesis was specific and confident, and a few of the suggested replacement strings drifted away from what the data model actually supports — "Someone with their own iPhone" for `device_client` reads like the device client owns the phone, but in this app device clients share the supervisor's phone with a PIN. Reading the source first beat patching to the brief.

## May 1 — Managed client double-rendered on the People list (commit `b505044`)

Add a managed client through People → "+" → "Someone I care for", and the new row appeared twice. One Firestore doc, two Core Data rows holding the same UUID. Confirmed by the user: deleting either row deleted the other, which is the fingerprint of the orphan-prune sweep doing its job after both rows lost their Firestore parent at once.

The shape of the bug, in `Dosely/Data/PersonRepository.swift`:

    try? await firestore.upsertPerson(fperson)
    return await context.perform { ... in
        let person = Person(context: context)   // always-insert
        ...
    }

The Firestore write fires the `/people` listener on this device. `SyncCoordinator.mirrorPeople` runs on a fresh background context, calls `FPerson.upsert`, finds nothing at this UUID, and inserts. Then the view-context block above runs and inserts ANOTHER row at the same UUID. The Person entity has no uniqueness constraint at the data-model layer (and Core Data doesn't enforce one without an explicit `<uniquenessConstraints>`), so both rows happily coexist. The People list iterated `[Person]` directly and rendered both.

Fix is the FPerson.upsert pattern lifted into both create paths: fetch by id, reuse the existing row if present, otherwise insert. Same change to `createDeviceClient`. ## May 13 — Medical ID save permission denied; encoded the error-collapse convention project-wide (commit `acd03c6`)

Save from EditMedicalIDView came back permission-denied and the UI rendered "Couldn't save. Check your connection and try again." Wrong copy — it's a rules rejection, not a network issue. Same error-collapse pattern that hit regenerateJoinCode in `c3018c2` and joinCareCircle in `1f6455c`. Fourth time. Time to stop patching one repository at a time and write the convention down.

The actual save failure came from the Codable-encode-then-override pattern in `upsertMedicalID`: `try encode(medicalID)` emits the FMedicalID with `updatedAt: Date(client)` as a Firestore Timestamp, then `payload["updatedAt"] = FieldValue.serverTimestamp()` overwrites it. Most SDK versions handle the override correctly, but the wire shape was a black box and the rule's `request.resource.data.updatedAt == request.time` check is unforgiving. Rewrote the payload as explicit dict construction with optional fields conditionally included. Same shape the rules tests have always verified; no more transient state between encode and ship.

But the real work is the convention. `FirestoreServiceError` already had four distinct cases — `.permissionDenied`, `.offline`, `.notFound`, `.unknown(String)`. The collapse was happening UPSTREAM, in repository catch chains that dropped the four cases into one or two domain-error buckets, and in UI catch sites that used a single generic "couldn't save" message regardless. The fix isn't bigger error types; it's the discipline to translate one-to-one all the way to the user-visible string.

Concrete changes:
- `MedicalIDRepositoryError` enum with the four-case shape mirroring `FirestoreServiceError`. `MedicalIDRepository.save` now `catch`es each FirestoreServiceError case and rethrows the matching domain case. No more `catch { throw .offline }`.
- `CareCircleEditError` gains `.unknown(String)`; regenerateJoinCode's previous "any other error → .offline" line — the very pattern this convention is meant to prevent — is gone. The detail string flows to `Logger` for post-hoc diagnosis.
- `EditMedicalIDView.save` branches on the four cases and shows distinct copy. "This action was denied. Sign out and back in if you think you should have access." reads very differently from "Check your connection."
- `FirestoreServiceError.map` writes every unmapped error to `Logger(subsystem: "com.medication.dosely", category: "firestore")` so a future investigation can use Console.app instead of attaching Xcode.
- Every catch site that maps Firestore errors carries `// Distinct error codes per error-collapse convention — see build_log April 30 phantom join code entry.` Grep-friendly for the next reviewer.
- `CLAUDE.md` gains an "Error-collapse convention" section under Coding conventions. Future Claude sessions inherit the rule via the loaded context — the cheapest possible enforcement.

Tests in `DoselyTests/ErrorCollapseConventionTests.swift` are type-level guards: `FirestoreServiceError.map` distinctly maps the three Firestore codes, `.unknown` carries the diagnostic detail string, non-Firestore domain errors don't collapse to `.offline`, and each repository's domain error enum carries distinct `.permissionDenied` vs `.offline` cases. A "let's simplify the error type" PR would fail at compile time first and these tests second.

Pattern-level note: shipping the same bug four times means the convention is missing, not that an engineer keeps making the same mistake. Patching one repo at a time is treating the symptom. Putting the rule in CLAUDE.md so it gets loaded into every future context, comment-marking every catch site so the rule travels with the code, and type-level tests so a future refactor fails loudly — all three, because any one of them on its own would have failed at least once already.

## May 28 — tightened `isOrphanFounder`, cleared every pre-existing rules-test failure, deployed

The helper at `firestore.rules:213` was over-permissive in the exact shape the May 27 "Discovered but not fixed" section diagnosed. The doc comment promised "founder anchor AND absence-of-membership"; the implementation only checked the founder anchor — the careCircle's `primarySupervisorPersonID` resolves to a Person doc whose `firebaseUID == request.auth.uid`. Every healthy primary supervisor satisfies that anchor as a side effect of being primary, so every `match` block that ORs `isOrphanFounder` in as an escape hatch was silently letting the primary do whatever the helper unlocked. The three rules-test failures named on May 27 all traced to the same root cause: `denies a sole supervisor from deleting their own Person doc` hits the Person delete rule at `firestore.rules:448`, whose `isOrphanFounder` short-circuits past the supervisorCount-and-primary-change checks; `denies the primary from leaving without atomically promoting a secondary` hits the same rule and the same short-circuit; `alerts cannot be deleted by any supervisor` hits the alert delete rule at `firestore.rules:641` which is gated exclusively on the helper. Three tests, one helper.

The fix adds the missing conjunct:

```
(
  !exists(/databases/$(database)/documents/userMemberships/$(request.auth.uid)) ||
  get(/databases/$(database)/documents/userMemberships/$(request.auth.uid)).data.careCircleID != circleID
)
```

## May 29 — Refill warnings: supply decrements on dose log, a detector fans out alerts, dose cards show days-left

`Medication.currentSupply` has been in the data model since the care-circle refactor, collected at add-time in `Step6SupplyView`, and never once decremented. The May 1 "Stubbed Refill alert removed" entry named exactly this gap — the field exists, nothing keeps it current, and Prompt 19's alert work was supposed to close it and didn't. This closes it.

**Part 1 — where the decrement plugs in.** `MedicationRepository.logDose` is the only dose-mutation path in the app — Today's take/skip, the dashboard's log-on-behalf, the TOOK_IT notification action, and `MissedDoseChecker`'s late/missed marking all funnel through it. There is no unlog or correct method; every tap inserts a fresh `DoseLog`. So the decrement plugs into `logDose`, and it rides in the SAME Firestore `WriteBatch` as the dose-log write (`FirestoreService.upsertDoseLog` now takes an optional `supplyChange`). Batching is the point: a dose log that lands without its supply change, or vice versa, would drift `currentSupply` off the real pill count and fire the refill alert at the wrong time — the exact bug class Part 1 warned against. Same batch means both commit or neither.

**Part 2 — decrement, restore, floor.** The supply delta is computed from the state TRANSITION for a scheduled time, not the new status in isolation: a `taken` log consumes one unit, a prior `taken` already counted means a re-tap (or a notification-then-in-app double log) adds nothing, and correcting away from `taken` restores one. `FieldValue.increment(±1)` does the actual write, server-side and race-safe. Floor at 0 is a local precheck — `increment` can't floor on the server, so `logDose` omits the decrement when the locally-known supply is already 0; no upward cap on a restore, because the user may have refilled in between and we never cap down what they know is correct. The new catch carries the error-collapse grep-marker; the batch maps Firestore errors distinctly via `.map`.

**Part 3 — `RefillSupplyCalculator`.** Pure value type: doses-per-day is the set weekday bits summed across every `DoseSchedule` (`daysOfWeek` is a Mon..Sun bitmask) over seven. Once-daily → 1, twice → 2, weekly (one one-day schedule) → 1/7 so supply × 7 days, a 3–4-day-a-week mask lands in between. A med with no schedules (as-needed) returns nil and is never "low" — an empty bottle of a PRN med isn't a refill signal. One simplification flagged: the model decrements by 1 per dose and treats `currentSupply` as doses-remaining, ignoring `pillsPerDose`; a 2-pill dose therefore over-states days-remaining (warns slightly late). Correct but conservative in the wrong direction — a Phase 2 item.

**Part 4 — `RefillAlertDetector`.** Mirrors `MissedDoseDetector` exactly — a plain class with `run(in:now:)` returning the attempted alert ids, idempotent via a deterministic id (`refill-{medID}-{todayISO}`, rendered in the calendar's zone like the weekly summary) so concurrent supervisor devices converge on one doc through `createIfAbsent`. It skips any med that already has an unacknowledged refill alert, so a med low for a week doesn't pile a fresh alert into the inbox every run — one pending refill alert per med until it's acknowledged. Run sites match the missed-dose detector: `SupervisorDashboardViewModel.load`, the dashboard foreground, and `TodayView`'s 5-minute timer + foreground. On a pure device-client surface the `isAnySupervisor` alert-create rule denies it and the `try?` swallows it (same as the missed-dose detector); on the shared device where the supervisor is the signed-in Firebase user — the common setup — it lands.

**Part 5 — the alert variant.** `Alert` is a single Core Data entity with a `type` string and a JSON `payloadJSON`, so the refill variant is a new `AlertType.refill` constant plus payload keys (medicationName, daysRemaining, currentSupply, personName, runOutDate) — no schema migration. `AlertID.refill` joins `missedDose` and `weeklySummary` as the third deterministic id.

**Part 6 — the surfaces.** `AlertsCard` gained a refill row: a `pills.fill` icon tinted `dsWarning`, a "Refill soon: {med}" headline above the "{days} days of supply remaining, refill before {date}" body, and the existing Acknowledge and Read-aloud affordances (the May 29 voice service reads it through `.alert(title:body:)` with no voice-side change). The other three alert types render exactly as before — the headline line is refill-only. `DoseCardView` gained a compact "{X} days left" capsule under the med name (`dsWarning` text on a 15%-tint `dsWarning` fill), shown only when `isLow` and suppressed entirely when days-remaining is nil (as-needed meds). It's a status badge, not a control — taps fall through to the card's existing expand gesture.

**Parts 7–9 — localization, tests, rules.** Four new `refill.*` keys in both bundles (Punjabi `// REVIEW NEEDED`), reusing the existing acknowledge and read-aloud keys rather than duplicating them. 24 new Swift tests — `RefillSupplyCalculatorTests` pins every schedule shape and the threshold boundary, `RefillAlertDetectorTests` drives the selection logic via the returned alert ids (low generates, above-threshold doesn't, as-needed is skipped, a pending alert suppresses a duplicate), and `MedicationRepositoryTests` gained the decrement/restore/floor/no-double-count cases. Suite 252 → 276, zero failures.

The rules needed one change, and it's load-bearing. The supply decrement is a medication-doc update, but the medication update rule was `isPrimary`-only — so a device client logging their own dose (which decrements in the same batch) would have been denied, and the whole batch with it, breaking device-client dose logging outright. Suite 276 → 291, zero failures. The reflection worth leaving: the static curated list is less impressive than a runtime openFDA-label scanner would be, and it is the right engineering for who this ships to. It's the same shape as the May 11 medical-ID decision to store list fields as a JSON-encoded array instead of a relational table: the boring choice was correct because the data is read whole and the failure modes of the clever version weren't worth the coverage they bought. The interesting engineering is rarely the right engineering for a four-medication grandmother.

## June 1 — Accessibility toggles: high-contrast mode and forced larger text

This closes "Accessibility toggles polish (Prompt 10 partial)" — the last of Prompt 10's original scope (text-size override, high-contrast mode, voice readout helper). Voice readout shipped May 29; this adds the two visual toggles, and the trailer line goes with them.

Two in-app toggles in a new Settings → Accessibility section, both defaulting off, both governed by one principle: **they only escalate, never reduce.** "Higher contrast" adds contrast on top of whatever iOS's "Increase Contrast" is doing; "Larger text" raises the Dynamic Type floor without capping iOS's larger settings. A trust line under them says exactly that — a user who's already tuned iOS for low vision shouldn't fear the app is overriding their work.

**The four-cell DSColors expansion.** The May 28 entry built the two-cell foundation: every token a `UIColor(dynamicProvider:)` branching on `userInterfaceStyle`. This adds the second axis — `accessibilityContrast` — so each token now resolves four cells: light/normal, light/high, dark/normal, dark/high. High contrast fires from iOS's trait OR the in-app `force_high_contrast` default (the `|| UserDefaults.standard.bool(...)` arm), so the two activation paths are independent and additive. Text pushes to pure black/white in high contrast; backgrounds in dark/high go to true `#000000`; semantic fills deepen.

One design rule I inverted from the prompt, for the same constraint that pins dsWarning. The prompt called for dark/high semantic fills to go *brighter* to lift off dark backgrounds — but white text sits on those fills (every action button), and the contrast suite this prompt extends pins white-on-fill ≥ 4.5:1. Dark/normal is already only ~5:1; a brighter fill *lowers* that and fails the floor. So dark/high fills go *deeper* (blue-700, amber-800, green-800, red-700), which is what actually raises white-text contrast on the buttons — the safety-critical metric the suite encodes. The 16 new four-cell contrast assertions prove every cell clears 4.5:1; the deviation is documented on the helper.

`dsWarning` keeps its pinned exception, now in two dimensions: light/normal == light/high (identical), because light amber sits at the white-on-fill boundary already (#D69E2E failed at 2.39:1) and deepening it clears contrast but loses the brand amber. Dark/high does deepen, to amber-800, where there's room. A test pins the light identity so a future "make warning adapt" change has to consciously re-check the floor.

**Wiring the toggles.** High contrast: the provider reads the `force_high_contrast` default, and `DoselyApp` folds the flag into the root `.id` so toggling forces a full rebuild and the colors re-resolve immediately — the same lever the language switch uses. (iOS's Increase Contrast is a real trait and re-resolves on its own; the `.id` exists only for the in-app default, since a UserDefaults change isn't a trait change and the cached per-view colors wouldn't otherwise refresh.) Larger text: `DoselyApp` applies `.dynamicTypeSize(.accessibility1...)` when on and a no-op `.xSmall...` floor when off — a `PartialRangeFrom` sets only a lower bound, raising small system sizes to the floor and leaving `.accessibility5` untouched. That floor-not-cap logic lives in a testable `AccessibilityScaling.effectiveSize` so "never reduces" is pinned without hosting the view.

**Tests, localization, rules.** 22 new tests: the four-cell contrast suite (16), the dsWarning light pin, the two `force_high_contrast` default cases — which also confirmed `resolvedColor(with:)` invokes the provider fresh rather than serving a stale per-trait cache (the thing that would have made the in-app toggle silently not work) — and the three larger-text floor cases. Suite 303 → 325, zero failures. Six new `accessibility.*` localization keys in both bundles (Punjabi `// REVIEW NEEDED`). Rules: audited the rules; no change needed — accessibility settings are local-only via `@AppStorage`, not synced.

On the May 28 bypass concern (a literal `.foregroundColor(.black)` somewhere that would defeat high contrast): that audit still holds and high contrast doesn't open a new gap. The sanctioned bypasses are white-on-fill (which *improves* as fills deepen), black dimming scrims (intentionally black regardless), adaptive system grays (disabled/inactive, low-stakes), and shadows. None defeat the extension, so there was no bypass to fix this round.

The reflection worth leaving: with this, Dosely's accessibility isn't a feature, it's a stack. Dynamic Type respected everywhere, semantic font modifiers everywhere, 48 pt tap targets everywhere via `DSSpacing.minTapTarget`, voice readout in English and Punjabi, dark-mode adaptive colors, and now both an iOS-respecting AND an in-app-override path for high contrast and larger text — six surfaces in concert. The design system turned out to be the load-bearing accessibility infrastructure: because every color and font routes through it, extending the system extended accessibility app-wide with no per-screen work. The toggles are the discoverability layer on top — for a grandparent who can't navigate iOS Settings to "Increase Contrast," the in-app switch is the one she can actually find. "We have an accessibility story" is the kind of claim that has to be backed by pieces actually in place; the pieces are now in place.

## June 6 — Demoting a secondary supervisor to a managed family member

The role graph has had one direction wired since May 1: a primary can promote a secondary to primary (and atomically demote themselves in the swap). The other direction — taking a caregiver's write access away without erasing them — had no entry point. `updatePersonRole`'s allowlist is `[deviceClient, managedClient]` and it rejects both ends of any supervisor transition by design: a supervisor↔client flip is privilege-significant and needs a cascade (`supervisorCount`, the `/userMemberships` index, the Firebase identity) that the generic role-flip path doesn't carry. `removePersonFromCircle` touches that cascade but deletes the Person outright, taking the dose-log history with it. This prompt ships the missing operation: convert a `secondary_supervisor` to a `managed_client`, keep the Person row and its history, strip the Firebase access.

**The repository and the batch (Parts 2/3).** `PersonRepository.demoteSupervisorToManagedClient(targetPersonID:actingSupervisorID:)` is modeled almost line-for-line on `promoteToPrimary`: a `context.perform` preflight that returns a `Plan` or throws a typed `PersonRepositoryError`, a per-case log in the catch, the single `FirestoreService` call, then the Core Data mirror — Firestore first, mirror skipped on failure. One deliberate divergence from the prompt's letter: it does the primary check *inline* (throwing the distinct `.notCurrentPrimary`) rather than calling `ensureCanWrite`. In this codebase `canWrite == isPrimary`, so `ensureCanWrite` would throw `.permissionDenied` for a secondary and pre-empt the `.notCurrentPrimary` the UI and the distinctness test both expect — `promoteToPrimary` already sidesteps it the same way, so the new method follows the proven sibling, not the instruction that predated it. `FirestoreService.applyDemotionToManagedClient` borrows `applyPrimaryAssignment`'s batch construction and `roleLogger` observability but diverges where the operation does: the membership write is a `deleteDocument`, not a `setData(merge:)` — a `managed_client` is not a Firebase member and the `/userMemberships` role enum doesn't even admit the value, so the index doc must go. `firebaseUID` is set to the empty string (the rules recognise a demoted account by `firebaseUID == ""`); `pinHash`/`pinSalt` use `FieldValue.delete()` so a future reader never sees an empty-string hash and misreads it as "PIN '' is valid."

**The rules, and the carve-out that does the actual work (Part 4).** The obvious move — add an `isDemotionToManagedClientBatch` helper and OR it onto the Person-update rule — would have enforced nothing. The first branch of that rule is `isPrimary(circleID) && id immutable`, which already authorises a primary to write *any* field, role included; a purely additive permissive branch cannot restrict what an existing permissive branch already allows. So branch (a) is now carved: it excludes the single transition `role -> managed_client when the pre-role was a supervisor flavour`, forcing the demotion through the constrained helper. The carve-out is surgical — it touches only supervisor→managed_client, a transition nothing in the app performed before this prompt, so the device↔managed flip, PIN resets, promotions, and `PrimaryRoleMigration` (all of which write roles through branch (a)) are untouched, confirmed by the full rules suite staying green. The helper itself pins the actor as primary, the pre-role as `secondary_supervisor` with a real UID, the new role/firebaseUID/PIN shape, the explicit `primarySupervisorPersonID != personID` guard (the parameter that would otherwise be unused — see the deploy note), and `getAfter(careCircle).supervisorCount == get(...).supervisorCount - 1`. That last clause is what makes the "count not decremented" attack fail at the Person write, since the careCircle's own branch (a) wouldn't stop a primary writing any count. The membership-delete-by-primary branch already existed (the `removePersonFromCircle` path), and the careCircle `supervisorCount`-1 is permitted by branch (a) for a primary and by branch (c) (`>= 1` floor) for any supervisor — both audited, both confirmed, no change needed to either. Five new emulator tests: the happy path, self-demotion (pre-role is primary, helper rejects), a secondary actor (`isPrimary` false), a managed_client target (no membership doc to delete, so the batch's delete is denied — the demotion path can't run against a non-secondary), and count-drift. Suite 94 → 99.

Deployed twice, on purpose. The first `firebase deploy --only firestore:rules` compiled with a warning — `personID` unused in the new helper — and rather than suppress it I gave the parameter a job: the explicit current-primary guard above, which is genuine defence-in-depth (the pre-role check already implies it, but a stale role during a promotion race shouldn't be the only thing standing between a primary and a roleless circle). Re-ran the emulator suite (still 99), re-deployed clean. **Published to `dosely-df5ca` / cloud.firestore at 2026-06-06T23:42:55Z** (the first, warning-carrying release was 23:41:55Z and is superseded). Recording both timestamps because the rules-deploy-lag class of bug has bitten this project repeatedly and the audit trail is the cheap insurance.

**The UI (Part 5).** A "Change role" card appears on `PersonDetailView` under a four-condition predicate, each load-bearing: the viewer is the primary (only the primary can demote); the target is a `secondary_supervisor`, checked against the constant not `isAnySupervisor`, so the legacy `"supervisor"` alias — which reads as primary — is excluded; the target isn't the viewer (self-demotion is blocked, and would orphan the primary slot); and the target isn't the current primary (the same orphan protection, against a target whose role string has gone stale). The predicate is a pure `static func shouldShowDemoteSection` so the four view tests pin every branch without hosting the view, matching the `EditMedicalIDView.shouldShowTargetPicker` pattern. The confirmation is a `sheet`, not the `.alert` the other confirmations use — a deliberate exception, because the prompt requires an in-flight loading state (a `ProgressView` in the Convert button, Cancel staying live) and a stock `.alert` dismisses itself on any button tap, so it can't host one. On success the detail view deliberately does *not* dismiss: the Person row is now a `managed_client`, so the demote card disappears and the managed↔device flip and medication sections render on the next pass — the supervisor watches the conversion land.

**Error-collapse (Part 6 plus the convention).** This is a privilege-*removing* action, which is exactly where collapsing `permissionDenied` into "check your connection" does the most damage — it sends a supervisor chasing a network fault when the truth is the server refused the write. So the flow gets its own `demoteErrorMessage` returning the `people.demote.error.*` set rather than reusing the shared `mapError`'s generic strings: distinct copy for `notCurrentPrimary`, `invalidDemotionTarget`, `permissionDenied`, `offline`, `notFound`, and `unknown`, each mapped one-to-one. The reflection worth leaving: Dosely's role model is now a graph, not a tree. A secondary can become primary; a secondary can become a managed patient. The asymmetry that remains is honest — a `managed_client` can't be lifted back to `secondary_supervisor` without a fresh join-code flow, because a supervisor needs a Firebase identity and you can't conjure one from a row that never had it; re-creating that account is materially different from preserving an existing one, and pretending otherwise would be the dishonest shortcut. That asymmetry is worth marking precisely because real families live it. The caregiver who set up the circle is sometimes the person whose own health declines into needing care, and the app should let the family move them across that line without deleting who they were — the dose logs from when they were the one keeping everyone else on schedule stay attached to their name. The graph has the edge that matters; the missing edge is the one the auth model, not the product, makes expensive.

## June 6 — Demote, fixed: the demoted account keeps its identity and lands on TodayView

Same-day follow-up to the demote feature above, and a partial walk-back of it. Josh demoted a secondary on one device and watched the demoted account, on its own device, get bounced to `CircleSetupView` — the "you have no family" screen — instead of `TodayView`. The cause was the original spec, not a slip in executing it: the demote batch cleared the Person's `firebaseUID` to `""` and DELETED `/userMemberships/{uid}`. `RemotePersonResolver` looks up the membership doc first; with it gone it returns `.notFound`, `AuthService` maps that to `currentPerson = nil`, and `AuthGate` routes a nil person to `CircleSetupView`. The original design modelled demotion only from the primary's side — "remove their access, keep their history" — and never walked the demoted user's own-device experience. It removed too much.

**The design clarification.** `managed_client` is no longer "a non-Firebase user, period." It now MAY carry a Firebase identity (a demoted supervisor, or a patient who signed up but doesn't want supervisor power) or MAY be fully passive (no identity; the supervisor logs for them). When it has an identity, it signs in and behaves like a device_client minus the PIN: views its own schedule, taps Take/Skip/Snooze on its own dose cards, authors its own dose logs. That is a CLAUDE.md role-definition rewrite plus a rules/repository widening, not just a bug patch — the bug report opened a door the original spec hadn't seen, the same way the May 28 dashboard reactivity gap turned into a live-update improvement rather than a one-line fix.

**What changed, file by file.** `PersonRepository.demoteSupervisorToManagedClient` no longer nils `firebaseUID` in the Core Data mirror (PINs still clear). `FirestoreService.applyDemotionToManagedClient` drops the `firebaseUID = ""` write and replaces the membership `deleteDocument` with an `updateData` that sets the membership's `role` to `managed_client` — same batch, same supervisorCount decrement, observability and comments rewritten to match. `firestore.rules`: `isDemotionToManagedClientBatch` loses its `firebaseUID == ""` clause (the UID is preserved now) while keeping every other invariant (actor is primary, target was a secondary with a real UID, no self/primary demotion, supervisorCount −1). ## June 6 — Patient-view hygiene: hide the debug toolbar and the "+" from clients, resize the banner

Josh signed in as the managed_client he'd demoted himself into and screenshotted the result. Three things were wrong on the patient's own `TodayView`: (1) the developer debug toolbar — "Test notification (30s)", "Schedule real dose (2 min)", "Clear drug cache" — sat across the top; (2) the Dosely banner rendered as a small white pill marooned on the dark page; (3) the "+" add-medication button was in the top-right, letting a managed_client open `AddMedicationFlow`. This is UI-only cleanup — no rules, no new features, strictly hiding existing affordances and one resize — and it is the direct sequel to the June 6 demote fix that first let a client reach this screen.

**The debug toolbar now gates on two independent layers.** It was already wrapped in `#if DEBUG` (the build-mode guard — it must never reach a RELEASE build). That stays, untouched. Added on top: a role guard, `Roles.isAnySupervisor(authService.currentPerson?.role)`, so even in a DEBUG build a device_client / managed_client never sees it. The two layers are belt-and-suspenders: `#if DEBUG` keeps it out of shipped builds; the role check keeps it out of a patient's view in the DEBUG builds a developer actually runs (which is exactly the build Josh was on when he demoted himself). The decision is a pure `static func shouldShowDebugToolbar(role:)` placed OUTSIDE the `#if DEBUG` so the test target can pin it in any configuration. The modifier already held `@EnvironmentObject var authService` (it reads `currentPerson` to schedule a real-dose verification), so no new property or call-site change was needed.

**The "+" is gated, and the rest of the surface was audited.** The add-medication button is wrapped in `if TodayView.shouldShowAddMedication(role:)` (a new static, delegating to `Roles.isAnySupervisor`) so it shows for both supervisor flavours and hides from clients — SwiftUI's `@ToolbarContentBuilder` takes a plain `if`, so the leading Settings item is untouched. (A secondary tapping "+" still hits the repository's primary-only `saveMedication` guard at save time; the UI fix just stops it appearing for clients, per the literal ask.) The audit of every other TodayView affordance: the Settings/person button stays open (Settings is for everyone; `SettingsSheet` does its own internal primary gating) — already correct; Take / Skip / Snooze on dose cards stay open for clients (`DoseCardView` defaults `showActions: true`; the read-only `showActions: isPrimary` variant lives on the supervisor dashboard, not here) — already correct, and the load-bearing piece of the June 6 fix, so a regression test pins it; "Learn more" opens a read-only `MedicationDetailView` — already correct, no editor; the Medical ID and "I need help" tiles are already client-gated (`isClientActor` / `isDeviceClient`) the other direction. The single leak in this file was the "+". I also deleted a stray `print("[UI-DEBUG] + tapped")` that had been left in the button's action.

**The banner resize.** The white badge had been hugging the small logo image while the outer frame stayed transparent — so it read as a centered ornament, not a header. Moved the white substrate (and the hairline border) to the full-width badge, bumped the logo `@ScaledMetric` from 40 to 64 (clamped 96), corners to `rLg`, inner padding `lg`/`md`, and an outer horizontal `lg` inset so the banner aligns with the dose-card column. Added `.accessibilityAddTraits(.isHeader)` so VoiceOver announces it as a heading. The white-on-dark badge remains the sanctioned non-token usage (DSColors category 5); the note is geometry-agnostic and still accurate, so it didn't change. The `imageName:` init and the text-wordmark fallback are preserved, so DoselyBannerTests stays green.

The reflection worth leaving: this is the third bug in two days whose root is "the spec never modelled this role's experience." The June 6 fix routed a demoted client to `TodayView` correctly at the AuthGate layer — but `TodayView` itself had been built on the unspoken assumption that its audience was a supervisor (with a person selector and developer affordances), because until June 6 no non-supervisor could ever reach it. The moment a Firebase-backed client could sign in, every supervisor-shaped affordance on that screen became a leak. The diagnosis is the same as the demote bug's (a transition has two sides) and the cure is mechanical: when a role gains access to a view it never had, sit in that role's seat and look. The fix is never the hard part; remembering to change seats is. Worth marking because the pattern has now recurred often enough to be a checklist item, not an insight.
